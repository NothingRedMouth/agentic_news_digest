# Регламент работы со стратегией ветвления Git Flow

Настоящий документ описывает правила работы с репозиторием проекта, основанные на классической модели Git Flow. Все разработчики обязаны придерживаться данной структуры.

---

## Основные ветки

### 1. `main` (или `master`)
* **Назначение:** Продакшн-версия системы. Код здесь всегда стабилен и готов к развертыванию.
* **Правило:** Прямые коммиты запрещены. Код попадает сюда только через Pull Request из веток `release/*` и `hotfix/*`.
* **Теги:** Каждый коммит в этой ветке помечается тегом версии (например, `v1.0.0`).

### 2. `develop`
* **Назначение:** Главная ветка разработки. Содержит последний готовый код для следующего релиза.
* **Правило:** Прямые коммиты запрещены (за исключением мелких правок конфигурации по согласованию). Слияние происходит через Pull Request из веток `feature/*`.

---

## Вспомогательные ветки

### 1. Ветки функций (`feature/*`)
Используются для разработки нового функционала или рефакторинга.

* **Откуда ответвляются:** `develop`
* **Куда вливаются:** `develop`
* **Именование:** `feature/PP-***-short-description` (где `PP-***` — ID задачи).
* **Жизненный цикл:**
  ```bash
  git checkout develop
  git pull origin develop
  git checkout -b feature/PP-***
  # ... работа над задачей, коммиты ...
  git push origin feature/PP-***
  # Создается Pull Request в ветку develop
  ```

### 2. Ветки релизов (`release/*`)
Используются для подготовки, стабилизации и тестирования очередного планового релиза.

* **Откуда ответвляются:** `develop`
* **Куда вливаются:** `main` и `develop`
* **Именование:** `release/vX.Y.Z` (например, `release/v1.1.0`).
* **Жизненный цикл:**
  ```bash
  git checkout develop
  git pull origin develop
  git checkout -b release/v1.1.0
  # ... исправление багов, найденных QA на стейджинге ...
  # Слияние в main:
  git checkout main
  git merge --no-ff release/v1.1.0
  git tag -a v1.1.0 -m "Release 1.1.0"
  # Слияние обратно в develop:
  git checkout develop
  git merge --no-ff release/v1.1.0
  # Удаление релизной ветки:
  git branch -d release/v1.1.0
  ```

### 3. Ветки исправлений (`hotfix/*`)
Используются для срочного исправления критических багов, обнаруженных на проде.

* **Откуда ответвляются:** `main`
* **Куда вливаются:** `main` и `develop`
* **Именование:** `hotfix/vX.Y.Z` (например, `hotfix/v1.1.1`).
* **Жизненный цикл:**
  ```bash
  git checkout main
  git pull origin main
  git checkout -b hotfix/v1.1.1
  # ... быстрое исправление бага ...
  # Слияние в main:
  git checkout main
  git merge --no-ff hotfix/v1.1.1
  git tag -a v1.1.1 -m "Hotfix 1.1.1"
  # Слияние в develop:
  git checkout develop
  git merge --no-ff hotfix/v1.1.1
  # Удаление ветки хотфикса:
  git branch -d hotfix/v1.1.1
  ```

## Правила хорошего тона (Best Practices)

1. **Частые коммиты:** Делайте коммиты логически атомарными (один коммит — одно законченное изменение).
2. **Свежая история:** Перед отправкой Pull Request всегда подтягивайте изменения из целевой ветки (`git pull origin develop`) и разрешайте конфликты локально.
3. **Защита веток:** Ветки `main` и `develop` должны быть защищены настройками репозитория (требование минимум 1 апрув перед слиянием).
4. **Удаление веток:** Удаляйте локальные и удаленные ветки `feature/*` сразу после их успешного слияния в `develop`.
